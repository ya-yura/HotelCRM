# 07 AI System

Sources: `01_product.md`, `03_domain.md`, `05_flows.md`

## 1. AI role

AI is an assistant layer that reduces typing, highlights risk, and speeds retrieval. It must not become the source of truth for reservations, room readiness, payment validity, or checkout completion.

## 2. AI use cases

### Booking parsing

- Input: pasted message, email text, messenger chat, OTA text export
- Output: structured reservation draft with confidence by field
- UX: user pastes text, sees extracted dates/guest/room request/price, confirms before save
- Risks: wrong date parsing, wrong guest count, wrong room type assumptions

### Daily assistant

- Input: today's local operational snapshot
- Output: short summary of arrivals, unpaid stays, risks, and suggested actions
- UX: dashboard brief with "review actions" button
- Risks: hiding important edge cases under over-summary

### Anomaly detection

- Input: reservations, room states, payments, sync failures, occupancy patterns
- Output: flagged anomalies such as possible double booking, suspicious unpaid checkout, room-state inconsistency
- UX: issue inbox with confidence and explanation
- Risks: alert fatigue, false positives

### Pricing hints

- Input: occupancy, upcoming demand, room type inventory, recent rates
- Output: suggested price adjustment range
- UX: optional suggestion in reservation pricing panel
- Risks: naive advice in local seasonal markets, overreliance

### Occupancy optimization

- Input: current room map, future arrivals/departures, room types, cleaning readiness, maintenance blocks, stay lengths
- Output: suggested room assignment or reassignment options that keep the room matrix dense and reduce fragmentation
- UX: show "best fit" assignment candidates during reservation creation, extension, and today operations
- Risks: optimizing for density while ignoring guest comfort, housekeeping load, or existing commitments

### Admin routine automation

- Input: today's arrivals, unpaid stays, dirty rooms, guest messages, shift notes, sync issues
- Output: prioritized admin to-do list, draft reminders, shift summary, and anomaly-based follow-ups
- UX: one operational inbox on the dashboard with suggested next actions and editable message drafts
- Risks: noisy recommendations, false urgency, overtrust in AI-produced summaries

### Guest communication

- Input: reservation details and predefined template intent
- Output: editable message draft for confirmation, arrival instructions, or payment reminder
- UX: generate editable text only, never auto-send in V1
- Risks: wrong tone, wrong language, wrong factual detail

### Search assistant

- Input: fuzzy human query like "family from yesterday in 203"
- Output: ranked candidate guest/reservation matches
- UX: natural language search box backed by hybrid retrieval
- Risks: returning wrong record as top result without clear context

## 3. AI architecture

### Where to use LLM

- parse unstructured booking text
- generate short daily summaries
- draft messages
- interpret natural-language search queries into structured filters
- explain why a room assignment option is recommended

### Where to use rules

- reservation conflict detection
- state transitions
- financial calculations
- room readiness logic
- sync conflict handling
- hard anomaly thresholds
- core room assignment constraints for occupancy optimization

### Where to use hybrid

- anomaly detection: rules for deterministic checks, LLM for explanation and clustering
- search: indexed retrieval first, LLM only to expand query meaning
- pricing hints: rules/statistics first, LLM only for explanation
- occupancy optimization: rules score valid room candidates first, LLM explains tradeoffs in human language

## 4. Feedback loop

- Users can correct parsed booking fields before save
- Corrections are stored as labeled examples for parser evaluation
- Daily assistant items can be marked useful / not useful
- False anomaly flags can be dismissed with reason
- Search result clicks provide ranking feedback
- Track per-use-case precision before enabling broader automation

## 5. AI safety boundaries

- AI cannot finalize reservation creation without user confirmation
- AI cannot override conflict checks
- AI cannot mark room ready, checked in, checked out, or paid
- AI cannot send guest messages automatically in V1
- AI cannot auto-commit room moves or density optimization without human confirmation
- AI suggestions must show source data or reasoning summary where possible
- Low-confidence extraction should fall back to manual review, not guess silently

## 6. Operational safeguards

- Use structured output schemas
- Store prompt and model version for auditability of AI-generated drafts where relevant
- Set confidence thresholds per field for booking parser
- Run PII minimization where possible when sending text to LLM
- Provide deterministic fallback UI if AI service is unavailable
